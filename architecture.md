# Webアプリケーションアーキテクチャ案

## 1. 目的

このプロジェクトでは、Flask と HTML/CSS/JavaScript を用いてポモドーロタイマーの Web アプリケーションを構築する。

添付 UI モックを前提に、以下を満たす設計を採用する。

- シンプルな画面構成で実装を始めやすいこと
- タイマーの進行が安定していること
- 将来的な機能追加に耐えられること
- ユニットテストを書きやすいこと


## 2. 設計方針

基本方針は、Flask を薄いサーバー層として使い、タイマーの進行制御はブラウザ側で行う構成とする。

- Flask は初回画面表示、設定値・実績データの取得、セッション完了の記録を担当する
- JavaScript はタイマー状態の保持、残り時間計算、UI 更新を担当する
- SQLite は設定値と実績データの保存を担当する

この方針を取る理由は、秒単位で変化するタイマー状態をサーバー側で逐次管理すると複雑性が上がるためである。永続化が必要なのは設定や完了実績であり、現在進行中のカウントダウン自体はクライアント側で処理する方が実装と保守の両面で有利である。


## 3. 全体アーキテクチャ

アプリケーションは以下の 4 層で構成する。

### 3.1 Web 層

Flask のルーティングと HTTP 入出力を担当する。

- HTML ページの返却
- API エンドポイントの提供
- リクエストの受け取りとレスポンス整形

この層には業務ロジックを極力置かない。ルート関数は薄く保ち、入力を受けてサービス層を呼び出し、結果を返すだけにする。

### 3.2 Service 層

アプリケーション固有の業務ロジックを担当する。

- タイマー状態遷移の判定
- セッション完了時の処理
- 今日の進捗集計
- 設定値の検証

この層のロジックは Flask や SQLite に直接依存しないように設計する。

### 3.3 Repository 層

データベースアクセスを担当する。

- 設定値の取得・更新
- セッション履歴の保存
- 日次集計の取得

SQLite への SQL 実行はこの層に閉じ込め、上位層からは抽象化されたインターフェース経由で利用する。

### 3.4 Frontend 層

ブラウザ上での状態管理と描画を担当する。

- 現在フェーズの表示
- 円形プログレスの描画
- 残り時間の更新
- 開始、停止、リセット操作
- API 通信

JavaScript 側でも、状態計算と DOM 操作は分離する。


## 4. 責務分担

### 4.1 サーバー側で持つもの

- 初期画面の配信
- 設定値の管理
- 完了セッションの記録
- 日次進捗の集計と返却

### 4.2 クライアント側で持つもの

- 現在の残り時間
- 現在のフェーズ
- 開始時刻、終了予定時刻
- 実行中タイマーの UI 表示

### 4.3 保存対象

永続化するのは次のような情報に限定する。

- 設定: 作業時間、短休憩、長休憩、長休憩に入るまでの回数
- セッション履歴: 完了日時、種別、集中時間
- 日次実績: 日付、完了回数、集中時間合計


## 5. タイマー制御方針

タイマーは「残り秒数を毎秒減算する」のではなく、「終了予定時刻との差分」で残り時間を求める方式を採用する。

理由は以下の通り。

- setInterval の誤差が累積しにくい
- 画面再読み込み後も状態復元しやすい
- バックグラウンド時のズレを補正しやすい

基本的な状態遷移は次の通り。

1. 作業フェーズを開始する
2. 終了時に完了実績を記録する
3. 規定回数に応じて短休憩または長休憩へ遷移する
4. 休憩終了後に次の作業フェーズへ戻る

この状態遷移判定は純粋関数として実装する。

- ルーティング

- 永続化処理
- フロントエンド状態管理
- フロントエンド描画処理


## 7. API 設計案

最低限、以下の API を用意する。

### 7.1 画面表示

- GET /
  - タイマー画面を返す

### 7.2 統計取得

- GET /api/stats/today
  - 今日の完了回数、集中時間合計を返す

### 7.3 設定取得・更新

- GET /api/settings
  - 現在の設定値を返す
- PUT /api/settings
  - 設定値を更新する

### 7.4 セッション記録

- POST /api/sessions
  - 作業セッション完了時に記録する

将来的に必要であれば、履歴取得や週次レポート API を追加する。


## 8. データモデル案

### 8.1 settings

- id
- work_duration_minutes
- short_break_minutes
- long_break_minutes
- long_break_interval

### 8.2 sessions

- id
- session_type
- focus_seconds
- completed_at

### 8.3 daily_stats

集計テーブルとして保持してもよいが、初期段階では sessions から都度集計してもよい。

- stat_date
- completed_count
- focus_seconds_total

初期実装では構造を簡単に保つため、daily_stats は後回しでもよい。データ量が少ない段階では sessions からの集計で十分である。


## 9. テスト容易性を高めるための追加設計

ユニットテストのしやすさを重視し、以下を明確に採用する。

### 9.1 ロジックを純粋関数化する

タイマーの状態遷移、進捗率計算、設定バリデーションは副作用のない関数として実装する。これにより、入出力だけを検証する単体テストが可能になる。

### 9.2 時刻取得を注入可能にする

サービス層で現在時刻を直接呼ばない。現在時刻は引数で受け取るか、Clock のような薄い抽象経由で渡す。これにより、日付境界や完了時刻のテストが安定する。

### 9.3 Flask を薄く保つ

ルート関数に集計や状態遷移を実装しない。Flask テストでは HTTP の入出力だけを検証し、業務ロジックの検証はサービス層のユニットテストに寄せる。

### 9.4 Repository を差し替え可能にする

サービス層は具体的な SQLite 実装ではなく、抽象化された Repository に依存する。テスト時はインメモリ実装やモックへ差し替えられるようにする。

### 9.5 Frontend でもロジックと描画を分離する

JavaScript は次のように分離する。

- timer-state.js: 状態遷移、残り時間、進捗率の計算
- timer-ui.js: DOM 更新、円形プログレス描画
- api.js: Flask API 呼び出し
- storage.js: localStorage の読み書き

この分離により、DOM に依存しないロジックを個別にテストできる。

### 9.6 アプリケーションファクトリを使う

Flask アプリは create_app 形式にする。これにより、本番設定とテスト設定を切り替えやすくなり、テスト用 DB やテストクライアントを扱いやすくなる。


## 10. テスト戦略

### 10.1 ユニットテスト

対象:

- タイマー状態遷移
- 長休憩判定
- 進捗率計算
- 設定値バリデーション
- 日次集計ロジック

### 10.2 リポジトリテスト

対象:

- SQLite への保存
- 設定値取得
- 当日分セッション集計

### 10.3 API テスト

対象:

- 正常系レスポンス
- 不正入力時のエラー応答
- セッション記録 API の入出力

### 10.4 フロントエンドテスト

対象:

- 残り時間計算
- 進捗率計算
- localStorage からの状態復元

タイマー UI の表示確認は最小限の結合テストに留め、ビジネスロジックの大半は関数単位で検証できる設計にする。


## 11. 将来拡張を見据えたポイント

この構成であれば、将来的に次の機能を追加しやすい。

- 通知音やブラウザ通知
- 週次、月次レポート
- タスク管理との連携
- ユーザー認証
- 複数端末同期
- PWA 化

特に、現在のタイマー状態をクライアント側、履歴と設定をサーバー側に分けているため、将来リアルタイム同期が必要になった場合も変更範囲を限定しやすい。


## 12. まとめ

本アプリでは、Flask を薄い Web 層として使い、タイマー制御は JavaScript、永続化は SQLite、業務ロジックはサービス層に分離する構成を採用する。

さらに、テスト容易性を優先して以下を徹底する。

- ロジックと副作用を分離する
- 時刻依存を注入可能にする
- ルートを薄く保つ
- 永続化を Repository 層へ隔離する
- フロントエンドでも状態計算と DOM 操作を分離する

この構成により、初期実装のしやすさと将来拡張性、ユニットテストのしやすさを両立できる。
